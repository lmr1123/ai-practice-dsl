# 大参林门店 AI 陪练产品解决方案

## 1. 项目定位

大参林门店 AI 陪练是一套面向门店员工的移动端训练工具，用于把总部标准话术、商品知识、会员权益和合规红线转化为“可学习、可小测、可对练、可评分、可复盘”的训练流程。

第一阶段不追求一次性做完整培训平台，而是先跑通两个高频场景：

1. 收银台引导办卡：提升新会员开通率，规范隐私异议处理和拒绝后的服务体验。
2. 益智营养类新品推荐：提升活动商品推荐能力，训练员工识别人群、探询需求、合规表达和处理异议。

核心原则：先把门店真实业务动作练准，再逐步接入更完整的 AI 对话、评分和管理看板。

## 2. 目标用户

| 用户 | 核心诉求 | 产品价值 |
| --- | --- | --- |
| 门店新员工 | 快速掌握标准开口、权益说明、商品卖点和合规红线 | 低成本完成上岗前训练 |
| 门店熟手员工 | 针对新品、会员活动、异议处理做短训 | 提升转化率和话术稳定性 |
| 店长/带教 | 不想反复一对一带教，希望看到员工薄弱项 | 用数据发现谁没练、谁没通过、哪里弱 |
| 区域/总部培训 | 希望统一门店话术、复制优秀经验、降低培训成本 | 把活动政策和标杆话术快速下发并追踪效果 |

## 3. 产品主链路

建议采用“训前学习 + 小测准入 + AI 对练 + 练后复盘”的四段式。

```text
选择训练场景
→ 微学习卡片
→ 训前小测
→ AI 顾客对练
→ 多维评分与复盘
→ 未通过维度重练
```

### 3.1 微学习卡片

目标：让员工先知道标准答案和红线，再进入对练。

形式：

- 每个场景 3-6 张卡片。
- 每张卡控制在 30-60 秒阅读。
- 内容包括：一句话开口、核心权益/卖点、标准探询问题、高频异议、禁止话术。

当前已完成：

- 办卡 5 张卡片。
- 新品 6 张卡片。
- 可编辑 HTML 源文件：`cards/index.html`。
- 手机端 MVP 已内置每个场景 3 张核心卡片。

### 3.2 训前小测

目标：防止员工在知识和红线不清楚时直接进入 AI 对练。

规则建议：

- 每个场景 3-5 题。
- 题型以情景选择题为主。
- 80 分或答对 2/3 题后进入对练。
- 错题要给出简短解释，并引导回到对应学习卡片。

示例：

```text
顾客说“不想留手机号，怕骚扰”，哪句最合适？
A. 我们绝对不会联系您
B. 手机号主要用于会员账户、积分和权益通知，不想接收营销通知可按规则关闭
C. 不留手机号就没法优惠
```

### 3.3 AI 顾客对练

目标：训练员工在真实顾客压力下完成标准动作。

AI 顾客需要由三层约束控制：

1. 角色卡：固定身份、需求、情绪、预算、顾虑。
2. 场景状态机：控制对话阶段，如开场、探询、异议、成交/拒绝。
3. 评分规则：固定关键动作、扣分项和一票否决项。

对练不是追求 AI 随机聊天，而是做到“角色稳定、表达可变、评分稳定”。

### 3.4 练后复盘

目标：让员工知道下一次怎么改，店长知道团队哪里弱。

复盘建议输出：

- 总分和是否通过。
- 维度分：开口、需求探询、推荐表达、异议处理、合规。
- 命中关键动作。
- 遗漏关键动作。
- 风险话术提醒。
- 优秀话术摘录。
- 下一次建议重练角色。

## 4. 场景设计

### 4.1 收银台引导办卡

业务目标：

- 提升新会员开通率。
- 降低员工对隐私问题、赶时间、优惠少等异议的处理波动。
- 规范拒绝后的礼貌收银，不因强推影响顾客体验。

训练重点：

- 30 秒开口。
- 本单权益优先。
- 免费、手机号、验证码、全国通用讲清楚。
- 隐私回应真实，不做绝对承诺。
- 顾客拒绝后最多轻量挽回一次。

关键红线：

- 不说“绝对不会联系您”。
- 不说“不涉及个人信息”。
- 不让顾客随便填手机号。
- 不使用公用会员卡。
- 不说“不办会员不能买”。

### 4.2 益智营养类新品推荐

业务目标：

- 提升活动商品推荐成功率和客单价。
- 训练员工从不同人群中识别推荐机会。
- 防止营养品销售中出现夸大功效和不合规承诺。

当前建议角色卡：

| 角色 | 推荐机会 | 高风险追问 |
| --- | --- | --- |
| 学生家长 | 学习压力、用眼、睡眠、成长发育 | 能不能长高、提高成绩、改善近视 |
| 上班族 | 熬夜、眼疲劳、注意力差 | 能不能提神、代替咖啡、熬夜没事 |
| 备考青年 | 复习疲劳、记忆力、专注力 | 考前能否见效、背书更快 |
| 老人家属 | 记忆力、视力、日常营养 | 慢病同服、预防脑梗 |
| 家庭囤货 | 活动价、全家适用 | 老人小孩都能吃、买两盒更有效 |

关键红线：

- 不承诺长高。
- 不承诺提高成绩。
- 不承诺改善近视。
- 不承诺治疗失眠。
- 老人慢病、长期用药必须建议咨询药师或医生。

## 6. AI 能力设计

### 6.1 知识库

知识库应包含：

- 会员权益政策。
- 当期活动规则。
- 商品资料卡。
- 标准话术。
- 高频异议和推荐回应。
- 风险话术和合规红线。
- 优秀员工话术案例。

要求：

- 知识库版本化，活动变更后可追溯。
- AI 只能基于知识库和角色卡回答，不自由编规则。
- 金额、券规则、适用条件等必须优先走结构化字段。

### 6.2 AI 顾客生成

AI 顾客不应完全随机生成，而应由角色卡驱动：

```json
{
  "role": "隐私敏感的新客",
  "goal": "快速结账，不想留手机号",
  "emotion": "不耐烦",
  "must_raise_objections": ["怕骚扰", "赶时间"],
  "can_accept_if": ["讲清免费", "说明手机号用途", "不强迫"],
  "forbidden_behavior": ["主动帮员工成交", "偏离收银台办卡场景"]
}
```

### 6.3 AI 评分

评分建议采用“规则评分 + 大模型解释”的组合。

规则评分适合：

- 是否出现一票否决话术。
- 是否提到手机号用途。
- 是否询问使用对象。
- 是否提到营养补充属性。
- 是否承诺治疗效果。

大模型解释适合：

- 话术是否自然。
- 异议回应是否有针对性。
- 推荐是否匹配客户需求。
- 是否过度推销。

## 8. 技术架构建议

### 8.1 当前 MVP

```text
静态 HTML/CSS/JS
本地场景数据
本地小测逻辑
本地关键词占位评分
```

适合快速验证流程和 UI，不依赖后端。

### 8.2 下一阶段架构

```text
移动端 H5
  ↓
训练业务 API
  ↓
场景/卡片/题库/记录数据库
  ↓
AI 对话服务
  ↓
企业知识库 + 角色卡 + 评分规则
```

核心接口建议：

- `GET /scenarios`：获取训练场景。
- `GET /scenarios/{id}/cards`：获取学习卡片。
- `GET /scenarios/{id}/quiz`：获取小测题。
- `POST /practice/session`：创建对练会话。
- `POST /practice/message`：发送员工话术并获取 AI 顾客回应。
- `POST /practice/score`：生成练后评分。
- `GET /training/records`：获取个人训练记录。
- `GET /dashboard/store`：获取门店训练看板。

## 9. 迭代计划

### 阶段 0：方案与内容准备，已完成

- 办卡原始话术整理。
- 新品图片资料整理。
- 两个陪练场景脚本。
- 结构化 JSON 场景配置。
- 微学习卡片源文件。
- 手机端静态 MVP。
- IP 形象初稿。

### 阶段 1：员工端闭环 MVP

目标：让门店员工可以完整完成一次训练。

范围：

- 优化手机端 UI。
- 完整接入 2 个场景的全部学习卡片。
- 小测题库扩充到每场景 5-8 题。
- 本地模拟多轮顾客追问。
- 生成基础练后报告。

交付标准：

- 员工能在 5-8 分钟完成一次训练。
- 训练结果可在本地或简单后端保存。
- 店长能看到至少“谁完成、是否通过”。

### 阶段 2：接入真实 AI 对练

目标：把占位对练升级为 AI 顾客自由对话。

范围：

- 接入大模型 API。
- 建立企业知识库。
- 配置角色卡和状态机。
- 接入 AI 评分。
- 支持文本输入，后续支持语音。

交付标准：

- AI 顾客角色稳定，不跑题。
- 同一评分规则下结果稳定。
- 练后报告能指出具体话术问题。

### 阶段 3：管理看板与运营

目标：从单人训练工具升级为门店训练运营工具。

范围：

- 店长看板。
- 区域看板。
- 训练任务下发。
- 活动版本管理。
- 优秀话术沉淀。
- 业务指标关联分析。

交付标准：

- 总部能按门店、区域、场景查看训练进度。
- 店长能定位员工薄弱项。
- 训练结果能辅助会员开通和新品动销复盘。

## 11. 近期推进建议

下一步建议按以下顺序推进：

1. 先把员工端 MVP 的 UI 和交互打磨到可试用。
2. 把 11 张微学习卡完整接入手机端，而不是每场景只放 3 张。
3. 每个场景扩到 5-8 道小测题。
4. 做本地多轮模拟对练，不急着接大模型。
5. 输出练后报告页。
6. 再接真实 AI 顾客和 AI 评分。

这样推进风险最低：先验证门店员工愿不愿意用、能不能学懂、流程是否顺，再投入 AI 接口和后台建设。
