# AI 陪练新场景提交、审核与上线工作流

> 适用对象：门店运营、商品运营、会员运营、培训负责人。
>
> 使用方式：业务只需在飞书表中提交“真实业务素材”，无需编写 AI 提示词、评分规则、开场语或完整剧本。提交后，系统生成场景方案，业务和合规确认后即可发布为新陪练场景。

## 0. 工作流总览

| 阶段 | 谁负责 | 输入 | 系统/技术动作 | 输出 |
| --- | --- | --- | --- | --- |
| 1. 业务提交 | 业务负责人 | 场景、顾客类型、正确做法、错误做法、权益/商品资料 | 校验字段完整性，提示缺失项 | 待生成的场景素材 |
| 2. 系统生成初稿 | 系统 | 业务提交表 | 生成学习卡片、顾客角色、对话目标、评分规则、红线和报告模板 | 场景方案初稿 |
| 3. 业务复核 | 业务负责人 | 场景方案初稿 | 标记需修改的事实性内容 | 业务确认版 |
| 4. 合规/药师复核 | 合规、药师或培训负责人 | 业务确认版 | 检查功效表达、用药边界、隐私表达、活动规则 | 合规通过版 |
| 5. 技术生成上线配置 | 技术/产品 | 合规通过版 | 写入场景配置、知识库、评分规则，生成移动端入口 | 待验收场景 |
| 6. 试练验收 | 业务 + 产品 + 试点门店 | 待验收场景 | 跑优秀话术、风险话术、边界问题测试 | 可上线场景 |
| 7. 上线复盘 | 店长、培训、总部 | 训练数据、低分报告、风险话术 | 优化资料库和评分规则 | 场景迭代版本 |

| 训练难度 | 单选 | 是 | 初级/进阶/挑战 |

| 最近更新时间 | 时间 | 自动 | 版本追踪 |

## 2. 状态流转规则

| 当前状态 | 触发动作 | 下一个状态 | 说明 |
| --- | --- | --- | --- |
| 草稿 | 业务补齐必填项并提交 | 待生成 | 业务只填素材，不写评分和 AI 脚本 |
| 待生成 | 系统生成场景方案 | 待业务复核 | 系统回填方案链接 |
| 待业务复核 | 业务确认事实内容准确 | 待合规复核 | 重点看金额、活动规则、商品卖点 |
| 待业务复核 | 业务发现资料不准 | 退回修改 | 退回给业务补充 |
| 待合规复核 | 合规/药师确认通过 | 待技术上线 | 商品、慢病、用药类必须经过此步 |
| 待合规复核 | 发现风险表达 | 退回修改 | 标注风险点 |
| 待技术上线 | 技术生成配置并发布测试环境 | 试练中 | 生成场景入口和评估报告 |
| 试练中 | 通过验收测试 | 已上线 | 可对门店开放 |
| 试练中 | 测试不通过 | 退回修改 | 回到资料或规则修正 |

## 3. 系统生成方案内容

业务提交后，系统生成的方案必须包含以下内容，方便业务看懂并确认。

| 生成内容 | 业务确认重点 |
| --- | --- |
| 场景名称和训练目标 | 是否符合本次业务任务 |
| 员工端学习卡片 | 权益、商品、政策、红线是否准确 |
| 顾客角色卡 | 顾客是否真实，是否覆盖主要销售机会 |
| AI 顾客开场语 | 是否像真实顾客，不要太标准化 |
| 对话推进逻辑 | 顾客拒绝、追问、接受的路径是否合理 |
| 员工必须命中点 | 是否覆盖业务要求 |
| 风险扣分项 | 是否覆盖禁止话术和合规红线 |
| 一票否决项 | 是否明确、可判断 |
| 实时点评口径 | 是否能指导员工当场改进 |
| 练后报告模板 | 是否能让店长看懂员工薄弱点 |

## 4. 上线前验收清单

| 验收项 | 通过标准 |
| --- | --- |
| 优秀话术测试 | 员工按标准做法表达时，系统能给高分，并指出命中点 |
| 风险话术测试 | 出现夸大功效、强制办卡、擅自调药等风险时，系统能明显扣分 |
| 顾客角色稳定 | 同一角色多次对话，核心顾虑和训练目标保持一致 |
| 对话自然度 | AI 顾客表达像真实顾客，不机械背题 |
| 评分稳定性 | 同类优秀话术评分接近，同类风险话术评分接近 |
| 合规边界 | 不生成不合规建议，不鼓励员工承诺疗效或替代医生判断 |
| 移动端可用 | 可进入场景、语音转写、对话、实时点评、结束报告 |
| 数据记录 | 训练记录能写入飞书或后台，方便后续复盘 |

## 5. 业务不会填时的引导方式

业务通常能说清楚“场景、顾客类型、好做法、坏做法、资料”，但很难写评分规则和完整脚本。因此提交表应尽量让业务选择或举例。

| 业务难填项 | 不建议问法 | 建议问法 |
| --- | --- | --- |
| 评分规则 | “请写出 100 分评分标准” | “员工必须做到哪几件事？哪些说法一定不能出现？” |
| AI 顾客角色 | “请设计 AI 客户 Prompt” | “门店最常遇到哪几类顾客？他们通常怎么说？” |
| 开场语 | “请写顾客开场白” | “顾客第一句通常会怎么问/怎么拒绝？” |
| 实时点评 | “请设计实时反馈策略” | “员工讲得好时好在哪里？讲差时通常差在哪里？” |
| 合规红线 | “请设计安全规则” | “哪些话员工说了会被投诉、误导或违规？” |

## 8. 业务提交后系统生成什么

| 系统生成内容 | 给业务看的说明 | 是否需要业务手写 |
| --- | --- | --- |
| 场景说明 | 员工进入对练前看到的训练背景 | 否 |
| 顾客角色卡 | AI 顾客的身份、需求、顾虑、情绪、拒绝点 | 否 |
| 对话目标 | 本场景员工必须练到的动作 | 否 |
| 顾客开场语 | AI 顾客第一句话 | 否 |
| 对话推进逻辑 | AI 顾客如何追问、拒绝、接受或结束 | 否 |
| 实时点评规则 | 员工说完后，系统如何提示做得好或待改进 | 否 |
| 评分规则 | 100 分制、多维评分、扣分项、一票否决 | 否 |
| 学习卡片 | 对练前的知识卡、话术卡、红线卡 | 否 |
| 练后报告 | 得分、命中点、遗漏点、风险话术、改进建议 | 否 |

## 9. 飞书填报表字段模板

### 9.1 基础信息

| 字段 | 填写方式 | 是否必填 | 业务填写示例 |
| --- | --- | --- | --- |
| 场景名称 | 简短描述训练场景 | 必填 | 收银台引导办卡 |
| 场景分类 | 单选：会员/商品/慢病/医保/服务/投诉/其他 | 必填 | 会员 |
| 适用岗位 | 多选：收银员/营业员/药师/店长/新人 | 必填 | 收银员、营业员 |
| 训练目标 | 用业务语言说明员工要练会什么 | 必填 | 结账时自然介绍会员权益，顾客拒绝时礼貌处理 |
| 场景发生位置 | 单选或填空 | 必填 | 收银台 |
| 训练难度 | 初级/进阶/挑战 | 必填 | 初级 |
| 推荐训练时长 | 1-5 分钟 | 选填 | 3 分钟 |
| 业务负责人 | 姓名/部门 | 必填 | 会员运营部 |
| 资料版本 | 活动期或政策版本 | 必填 | 2026 年 5 月会员活动 |

### 9.2 顾客类型

| 字段 | 填写方式 | 是否必填 | 业务填写示例 |
| --- | --- | --- | --- |
| 主要顾客类型 | 勾选或填写 3-6 类 | 必填 | 赶时间、不想留手机号、觉得优惠小、不常来 |
| 顾客常见原话 | 填 3-8 句真实门店原话 | 必填 | “我赶时间，快点结账”“不想留手机号，怕骚扰” |
| 顾客真实顾虑 | 用短句说明 | 必填 | 怕被打扰、觉得办卡麻烦、觉得优惠不明显 |
| 顾客可接受条件 | 填员工做到什么后顾客可能接受 | 选填 | 讲清免费办理、本单能省钱、不会强推 |
| 顾客不可接受点 | 填哪些行为会让顾客反感 | 必填 | 强行要求验证码、一直追问、说不办不能优惠 |

### 9.3 员工正确做法

| 字段 | 填写方式 | 是否必填 | 业务填写示例 |
| --- | --- | --- | --- |
| 员工必须做到 | 填 3-8 个动作，不用写完整话术 | 必填 | 先确认是否有会员；说明免费办理；讲清本单优惠；顾客拒绝后正常收银 |
| 推荐表达重点 | 填权益、卖点、服务价值 | 必填 | 新人券、会员价、积分、会员日、全国通用 |
| 异议处理做法 | 填常见拒绝如何回应 | 必填 | 解释手机号用途，不做绝对承诺；觉得优惠小时帮顾客算本单优惠 |
| 成交或结束动作 | 填最终应如何收口 | 必填 | 顾客接受则协助开卡；顾客拒绝则礼貌结束并继续收银 |
| 优秀话术样例 | 有就填，没有可空 | 选填 | “不办也没关系，我先帮您正常结账。” |

### 9.4 错误做法与合规红线

| 字段 | 填写方式 | 是否必填 | 业务填写示例 |
| --- | --- | --- | --- |
| 明确禁止话术 | 填 3-10 句 | 必填 | “不办会员不能买”“绝对不会联系您”“随便填个手机号” |
| 高风险行为 | 填员工不能做什么 | 必填 | 强迫顾客办卡、夸大商品效果、忽视禁忌人群 |
| 一票否决项 | 出现即判不通过 | 必填 | 承诺疗效、虚假优惠、强制办卡、误导用药 |
| 差评话术样例 | 有就填，没有可空 | 选填 | “你不办卡就没有优惠，别人都办了。” |

### 9.5 企业权益或商品资料库

| 字段 | 填写方式 | 是否必填 | 业务填写示例 |

## 10. 示例一：新品推荐

### 10.1 业务填报示例

| 字段 | 示例内容 |
| --- | --- |
| 场景名称 | 益智营养类新品推荐 |
| 场景分类 | 商品 |
| 适用岗位 | 营业员、药师、新人 |
| 训练目标 | 员工能根据顾客人群和需求，自然推荐新品，并避免夸大功效 |
| 场景发生位置 | 货架区、收银台、顾客咨询区 |
| 训练难度 | 进阶 |
| 推荐训练时长 | 4 分钟 |
| 资料版本 | 2026 年 5 月益智营养类活动 |

| 顾客类型 | 顾客常见原话 | 真实顾虑 | 可接受条件 | 不可接受点 |
| --- | --- | --- | --- | --- |
| 学生家长 | “孩子最近学习压力大，有没有补脑的？” | 怕没效果，怕乱补 | 先问年龄、学习压力、睡眠、用眼情况，再说明营养补充边界 | 承诺提高成绩、承诺长高 |
| 备考青年 | “最近背书很累，想买点增强记忆的。” | 想快速见效 | 说明营养支持，不承诺短期见效 | 说吃了马上记忆变好 |
| 上班族 | “我经常熬夜、眼睛累。” | 想缓解疲劳 | 结合用眼、作息和营养补充建议 | 暗示可以替代休息或治疗眼病 |
| 老人家属 | “老人记忆力差，能不能吃？” | 担心慢病同服 | 建议先了解慢病和用药情况，必要时咨询药师 | 直接说老人都能吃 |

| 员工必须做到 | 推荐表达重点 |

## 14. 建议的业务审核流程

1. 业务填写飞书材料表。
2. 系统生成场景方案初稿。
3. 业务查看“方案预览”：场景说明、顾客角色、对话目标、红线、评分重点。
4. 业务只修改事实性内容：权益金额、商品卖点、禁忌、活动范围。
5. 培训或合规负责人确认。
6. 提交生成上线场景。
7. 试点门店练习 1 周，查看低分原因和风险话术。
8. 根据真实对练数据优化资料库。

## 15. 给业务的填写原则

- 不需要写完整剧本，写“员工应该怎么做”即可。
- 不需要设计评分，系统会根据正确做法和错误做法生成。
- 不需要写 AI 顾客，系统会根据顾客类型生成。
- 金额、活动规则、适用范围必须写准确。
- 商品功效必须使用合规表达，避免“治疗、保证、一定、马上见效”等绝对化词。
- 不确定的医学问题，应写“转药师或建议就医”，不要让系统自由判断。
